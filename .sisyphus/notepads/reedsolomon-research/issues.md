# Research Issues/Gotchas

## Issue 1: They don't have dedicated AVX512 for GF(2^8) in the way I initially expected

**Initial assumption**: They would have AVX512-optimized GF(2^8) with wide registers
**Reality**: Their "AVX512" path uses VGF2P8AFFINEQB which operates on 64 bytes per iteration but the instruction itself is the same width as AVX2's 32-byte version. The speedup comes from operating on 2x data per instruction.

**Lesson**: For GF(2^8), GFNI provides ~2x throughput over PSHUFB because it eliminates the split-table-XOR step.

## Issue 2: The README claims ~28 GiB/s but this is for GF(2^8) not larger fields

**Data point**: README shows "GFNI MB/s" column with numbers like 241,747 MB/s for encoding
**Clarification**: This is processing 256-bit (32-byte) lanes per cycle with GFNI, achieving ~28 GiB/s throughput for small matrices

**Lesson**: Their architecture is highly optimized for GF(2^8) specifically - don't extrapolate these numbers to GF(2^32).

## Issue 3: They use a completely different polynomial system for GF(2^16)

From `leopard.go`:
- GF(2^16) uses polynomial **0x1002D** (not the CRC32 polynomial)
- This is the "novel polynomial basis" from the paper they cite
- The matrices `gf2p811dMulMatrices16` encode multiplication in this basis

**Lesson**: Different polynomials require different multiplication architectures. The CRC32 polynomial 0x100000001B has special properties for carryless multiplication.

## Issue 4: They generate assembly via Avo (Go assembly generator)

The files `galois_gen_amd64.s` and `galois_gen_nopshufb_amd64.s` are generated by `go generate` from `_gen/gen.go`. 

**Implication**: The GFNI implementations are machine-generated from a table-driven approach.

## Issue 5: No VPCLMULQDQ usage found

Despite searching for CRC32, PCLMUL, etc., the library uses:
- PSHUFB for GF(2^8) (LUT-based)
- VGF2P8AFFINEQB for GFNI (native GF(2^8))
- No carryless multiplication for larger fields

**Lesson**: Their approach relies entirely on lookup tables or native GFNI instructions. For GF(2^32) with poly 0x100000001B, we'll need a different approach since no native instruction exists.

## Issue 6: VGF2P8AFFINEQB only works for GF(2^8)

The instruction encoding:
```
VGF2P8AFFINEQB imm8, src, dst
```
- Works on 8-bit elements only
- The polynomial is encoded in the immediate (imm8) 
- Cannot be used for 16, 32, or 64-bit field elements

**Lesson**: For GF(2^32), Intel provides CRC32 (for reduction) and VPCLMULQDQ (for multiplication), but no single-instruction field multiplication.
